# [منطق] عدم وجود fallback mechanism در context_builder برای VerifyConfig

## Raw Idea

## 📋 شرح ناسازگاری
در pipeline `ai_llm` یک ناسازگاری منطقی پیدا شد:

context_builder از WatchedProject.verify_v6_config به‌عنوان ورودی استفاده می‌کند، اما اگر این config وجود نداشته باشد یا ناقص باشد، هیچ fallback به default config پیش‌بینی نشده است. همچنین clamp مقادیر out-of-range ذکر شده اما منطق آن مشخص نیست.

## 💥 پیامد (impact)
اگر verify_v6_config None باشد یا فیلدهای ضروری را نداشته باشد، کل verify run ممکن است با خطا مواجه شود یا با مقادیر پیش‌فرض نادرست اجرا شود.

## 🛠 پیشنهاد رفع اولیه
اضافه کردن یک default VerifyConfig در context_builder که در صورت نبود config از WatchedProject استفاده شود، و پیاده‌سازی explicit validation با raise exception برای فیلدهای حیاتی.

## 🤔 چرا مهم است
coherence issue یعنی دو بخش کد فرض‌های ناسازگار دارند — معمولاً نشانه‌ی refactor ناتمام یا feature flag rot است. این کلاس bug ها در test معمولی پیدا نمی‌شوند چون unit test ها در silo اجرا می‌شوند.
