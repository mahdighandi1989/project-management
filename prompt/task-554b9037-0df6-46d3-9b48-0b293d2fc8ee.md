# [منطق] عدم وجود rate limiting و protection در برابر chunked upload abuse

## Raw Idea

## 📋 شرح ناسازگاری
در pipeline `auth` یک ناسازگاری منطقی پیدا شد:

سرویس oversight_upload_session از chunked و resumable upload پشتیبانی می‌کند، اما هیچ محدودیتی برای تعداد chunkها، نرخ آپلود، یا حجم کل فایل‌های همزمان وجود ندارد. همچنین از asyncio.Lock برای per-session lock استفاده می‌کند، اما این lock از race condition در سطح session جلوگیری می‌کند، نه از حملات DoS یا resource exhaustion.

## 💥 پیامد (impact)
یک مهاجم می‌تواند هزاران سشن آپلود با chunkهای کوچک ایجاد کند و دیسک سرور را پر کند (disk flood). همچنین می‌تواند با ارسال chunkهای تکراری یا ناقص، state machine را به حالت inconsistent ببرد.

## 🛠 پیشنهاد رفع اولیه
rate limiting بر اساس IP و user اضافه کنید. محدودیت برای حداکثر تعداد سشن‌های همزمان per user. محدودیت برای حداکثر حجم کل فایل‌های در حال آپلود. همچنین validation برای ترتیب و完整性 chunkها (مثلاً checksum) اضافه شود.

## 🤔 چرا مهم است
coherence issue یعنی دو بخش کد فرض‌های ناسازگار دارند — معمولاً نشانه‌ی refactor ناتمام یا feature flag rot است. این کلاس bug ها در test معمولی پیدا نمی‌شوند چون unit test ها در silo اجرا می‌شوند.
