# [منطق] عدم وجود logging امنیتی و audit trail

## Raw Idea

## 📋 شرح ناسازگاری
در pipeline `auth` یک ناسازگاری منطقی پیدا شد:

هیچ‌کدام از components به logging امنیتی اشاره‌ای ندارند. database.py فقط از logging برای خطاهای اتصال استفاده می‌کند. inspector_session لاگ‌های بررسی شده را ذخیره می‌کند اما این لاگ‌ها برای audit trail امنیتی (مانند لاگین، logout، تغییر permission) نیستند. oversight_upload_session نیز لاگ عملیات آپلود را ثبت نمی‌کند.

## 💥 پیامد (impact)
در صورت وقوع حادثه امنیتی (مانند دسترسی غیرمجاز یا تغییر داده)، هیچ traceability وجود ندارد. نمی‌توان تشخیص داد چه کسی، چه زمانی و چه عملیاتی انجام داده است.

## 🛠 پیشنهاد رفع اولیه
یک سرویس logging امنیتی اضافه کنید که تمام عملیات حساس (ایجاد سشن، آپلود فایل، تغییر وضعیت) را با timestamp, user_id, IP, action_type, status ثبت کند. این لاگ‌ها باید immutable و در یک مکان امن ذخیره شوند.

## 🤔 چرا مهم است
coherence issue یعنی دو بخش کد فرض‌های ناسازگار دارند — معمولاً نشانه‌ی refactor ناتمام یا feature flag rot است. این کلاس bug ها در test معمولی پیدا نمی‌شوند چون unit test ها در silo اجرا می‌شوند.
