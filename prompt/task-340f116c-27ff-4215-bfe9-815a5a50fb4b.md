1. یک Endpoint جدید `POST /analysis/run-dependency-check` (یا مشابه) باید با موفقیت یک تحلیل وابستگی را آغاز کرده و یک `DependencyReportSchema` معتبر را برگرداند. _(verify: api_response)_
2. گزارش‌های وابستگی ذخیره‌شده در پایگاه داده (مدل `DependencyReport`) باید شامل حداقل یک وابستگی شناسایی‌شده، وضعیت آن (مثلاً 'outdated') و یک 'دستورالعمل به‌روزرسانی' قابل فهم باشد. _(verify: backend_test)_
3. Endpoint `GET /analysis/dependency-reports/{report_id}` باید جزئیات کامل یک گزارش وابستگی را بر اساس `report_id` برگرداند. _(verify: api_response)_
4. سیستم باید قادر به شناسایی وابستگی‌های داخلی (مانند importهای پایتون یا ماژول‌های جاوااسکریپت) و وابستگی‌های خارجی (مانند پکیج‌های `requirements.txt` یا `package.json`) باشد. _(verify: backend_test)_
5. قابلیت زمان‌بندی بررسی‌های دوره‌ای وابستگی (مشابه `update_schedule` در `backend/app/api/routes/analysis.py`) باید پیاده‌سازی شود و به درستی کار کند. _(verify: backend_test)_
6. رابط کاربری فرانت‌اند (مثلاً در `frontend/src/app/project/[id]/page.tsx`) باید قادر به نمایش لیست گزارش‌های وابستگی و جزئیات هر گزارش باشد. _(verify: ui_interaction)_

## Task Steps

### Step 1: طراحی و پیاده‌سازی سیستم Dependency Graph برای ثبت وابستگی‌های بین اجزای پروژه
**Status:** `pending` (0%)
**Scope:** این مرحله شامل طراحی یک ساختار داده (مثلاً گراف جهت‌دار) برای ذخیره‌سازی وابستگی‌های بین تمام اجزای پروژه (فایل‌ها، ماژول‌ها، APIها، کتابخانه‌ها، توکن‌ها، کانفیگ‌ها) است. باید یک فایل مرکزی (مثلاً dependency_graph.json یا یک دیتابیس سبک) ایجاد شود که هر جزء و لیست وابستگی‌هایش را نگه دارد. همچنین باید یک API یا تابع برای به‌روزرسانی این گراف هنگام تغییرات فراهم شود. خارج از این مرحله: خودکار تشخیص وابستگی‌ها از کد (این کار در مرحله بعدی انجام می‌شود). نکته حیاتی: ساختار باید مقیاس‌پذیر باشد و امکان جستجوی معکوس (reverse dependency) را فراهم کند.
**Excerpt:**
```
یه قسمتی باید باشه که تمام این موارد رو که وقتی مثلاً هر چیزی انجام می‌شه بررسی بکنه که مثلاً چیزای دیگه‌شم باید باهاش به روزرسانی بشه و اینو مثلاً هر چند وقت یک بار توضیحات کامل بهش بده و با دسترسی‌هایی که مثلاً به چیز داره، توکن و این توکن‌های گیت‌هاب و هرچی که داره با این دسترسی‌ها بررسی بکنه همه وابستگی
```

### Step 2: پیاده‌سازی اسکنر خودکار برای شناسایی وابستگی‌های موجود در کد و مستندات
**Status:** `pending` (0%)
**Scope:** این مرحله شامل نوشتن یک اسکنر (مثلاً به صورت اسکریپت CLI یا تابع) است که کدبیس پروژه را اسکن کرده و وابستگی‌های بین اجزا را به صورت خودکار استخراج کند. این اسکنر باید importها، فراخوانی‌های API، استفاده از توکن‌ها، ارجاع به فایل‌های کانفیگ و هر نوع وابستگی صریح را تشخیص دهد. خروجی این اسکنر باید با ساختار Dependency Graph مرحله قبل سازگار باشد. خارج از این مرحله: تشخیص وابستگی‌های ضمنی یا runtime. نکته حیاتی: اسکنر باید قابل اجرا به صورت دوره‌ای (scheduled) باشد.
**Excerpt:**
```
مثلاً اگه یه چیزی رو ارتقا دادم قسمت‌های دیگر هم باید بگه که این قسمت باید با این موارد تطبیق داده بشه و دستوراتشو بده. یه جورایی مثل همین وریفای کردن منتها بر اساس وابستگی بین اجزای خود سیستمه.
```

### Step 3: ساخت سیستم اعلان و هشدار برای تغییرات وابسته (Dependency Change Notifier)
**Status:** `pending` (0%)
**Scope:** این مرحله شامل پیاده‌سازی یک سیستم است که پس از هر تغییر در یک جزء (مثلاً commit در گیت یا تغییر در فایل)، لیست تمام اجزای وابسته به آن را از Dependency Graph استخراج کرده و یک هشدار/اعلان تولید کند. این اعلان باید شامل: نام جزء تغییر یافته، لیست اجزای متأثر، نوع وابستگی و دستورالعمل پیشنهادی برای به‌روزرسانی باشد. خروجی می‌تواند به صورت لاگ، ایمیل، یا issue در گیت‌هاب باشد. خارج از این مرحله: اجرای خودکار تغییرات (فقط اعلان). نکته حیاتی: باید از توکن‌های گیت‌هاب برای دسترسی به مخزن استفاده کند.
**Excerpt:**
```
با دسترسی‌هایی که مثلاً به چیز داره، توکن و این توکن‌های گیت‌هاب و هرچی که داره با این دسترسی‌ها بررسی بکنه همه وابستگی
```

### Step 4: ایجاد گزارش دوره‌ای وضعیت وابستگی‌ها (Dependency Health Report)
**Status:** `pending` (0%)
**Scope:** این مرحله شامل تولید یک گزارش جامع به صورت دوره‌ای (مثلاً هفتگی) است که وضعیت تمام وابستگی‌های پروژه را نشان دهد: کدام اجزا به‌روز هستند، کدام نیاز به به‌روزرسانی دارند، کدام وابستگی‌ها قدیمی یا شکسته شده‌اند. این گزارش باید بر اساس اسکن خودکار و مقایسه با آخرین وضعیت ثبت‌شده در Dependency Graph تهیه شود. خروجی می‌تواند یک فایل Markdown یا HTML باشد. خارج از این مرحله: رفع خودکار مشکلات. نکته حیاتی: گزارش باید شامل توضیحات کامل و دستورات عملی برای هر مشکل باشد.
**Excerpt:**
```
اینو مثلاً هر چند وقت یک بار توضیحات کامل بهش بده و با دسترسی‌هایی که مثلاً به چیز داره، توکن و این توکن‌های گیت‌هاب و هرچی که داره با این دسترسی‌ها بررسی بکنه همه وابستگی
```

### Step 5: یکپارچه‌سازی با Git Hooks برای تشخیص خودکار تغییرات
**Status:** `pending` (0%)
**Scope:** این مرحله شامل تنظیم Git hooks (مخصوصاً post-commit و pre-push) است که پس از هر commit، اسکنر و سیستم اعلان را به صورت خودکار اجرا کنند. همچنین باید یک hook برای بررسی اینکه آیا تغییرات فعلی وابستگی‌های شکسته ایجاد می‌کند یا خیر (مانند CI/CD ساده) اضافه شود. خارج از این مرحله: استقرار روی سرور CI/CD. نکته حیاتی: hooks باید غیرمسدودکننده باشند (فقط هشدار دهند، مانع commit نشوند مگر در موارد بحرانی).
**Excerpt:**
```
مثلاً اگه یه چیزی رو ارتقا دادم قسمت‌های دیگر هم باید بگه که این قسمت باید با این موارد تطبیق داده بشه و دستوراتشو بده.
```

### Step 6: طراحی و پیاده‌سازی واسط کاربری (CLI/Dashboard) برای مشاهده و مدیریت وابستگی‌ها
**Status:** `pending` (0%)
**Scope:** این مرحله شامل ایجاد یک رابط کاربری ساده (ترجیحاً CLI در ابتدا و در صورت نیاز یک داشبورد وب ساده) است که کاربر بتواند: 1) وابستگی‌های یک جزء خاص را ببیند، 2) اجزای متأثر از یک تغییر را مشاهده کند، 3) گزارش‌های دوره‌ای را مرور کند، 4) به صورت دستی وابستگی جدید اضافه یا ویرایش کند. خارج از این مرحله: طراحی UI پیشرفته. نکته حیاتی: CLI باید با دستورات ساده و مستندات داخلی (help) همراه باشد.
**Excerpt:**
```
یه قسمتی باید باشه که تمام این موارد رو که وقتی مثلاً هر چیزی انجام می‌شه بررسی بکنه
```

### Step 7: نوشتن تست‌های واحد و یکپارچه‌سازی برای کل سیستم
**Status:** `pending` (0%)
**Scope:** این مرحله شامل نوشتن تست‌های جامع برای: 1) صحت Dependency Graph (اضافه، حذف، جستجو)، 2) اسکنر خودکار (با نمونه کدهای آزمایشی)، 3) سیستم اعلان (با mock کردن گیت‌هاب API)، 4) Git hooks (با شبیه‌سازی commit). تست‌ها باید حداقل 80% پوشش کد را داشته باشند. خارج از این مرحله: تست‌های end-to-end روی مخزن واقعی. نکته حیاتی: تست‌ها باید مستقل از محیط اجرا باشند.
**Excerpt:**
```
مشکلی که الان هست اینه که این پروژه آپشن‌ها و گزینه‌های خیلی زیادی پیدا کرده که خیلی‌هاش یا قدیمیه و کار نمی‌کنه و خیلی‌هاش به اصطلاح نیاز داره که اصلاً بدونم چی بوده، برای چیه و هر سری باید بهش سر بزنم.
```
