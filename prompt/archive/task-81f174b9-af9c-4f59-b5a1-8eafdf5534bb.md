  async def configure_pre_action_models(
      project_id: str,
      model_ids: List[str],
      project_service: ProjectService = Depends(ProjectService)
  ):

      Configures the AI models to be used for pre-push/pre-run actions for a project.
      """

  ```
- `backend/app/services/project_service.py` — `ProjectService` — این فایل deep-read نشده — مجری باید مسیر را خود تأیید کند. این مکان برای پیاده‌سازی منطق کسب‌وکار برای به‌روزرسانی تنظیمات مدل‌های پیش از عملیات پروژه است.
  ```python
  class ProjectService:
      def __init__(self, db: AsyncSession = Depends(get_db)):
          self.db = db
          self.ai_manager = AIManager() # Assuming AIManager can be instantiated or injected
  
      async def update_pre_action_config(self, project_id: str, model_ids: List[str]) -> Optional[Project]:
          project = await self.db.get(Project, project_id)
          if not project:
              return None
  
          # Validate model_ids (conceptual validation using AIManager/AIProfile)
          valid_model_ids = [mid for mid in model_ids if await self.ai_manager.is_model_available(mid)] # Example validation
  
          project.pre_action_selected_model_ids = valid_model_ids
          await self.db.commit()
          await self.db.refresh(project)
          return project
  ```

## 🔍 Context و وضعیت فعلی
کاربر درخواست کرده است که یک "نقطه اتصال پیش از پوش/اجرا" (pre-push/pre-run connection point) در گردش کار توسعه پروژه تعریف و یکپارچه‌سازی شود. هدف این است که یک مکانیزم (مانند یک هوک گیت، یک مرحله پیش از CI/CD، یا یکپارچه‌سازی رابط کاربری وب) پروژه را قبل از اینکه به گیت‌هاب پوش شود یا به صورت محلی اجرا گردد، متوقف کند. این توقف باید در مرحله حیاتی «پیش از پوش/اجرا» اتفاق بیفتد تا اطمینان حاصل شود که فرآیند بازبینی و انتخاب مدل‌ها به موقع انجام می‌شود و از بروز مشکلات احتمالی در مراحل بعدی جلوگیری می‌کند.

به‌طور خاص، کاربر می‌خواهد "توی صفحه گیت‌هاب، اونجایی که پروژه حالا به صورت محلی ساخته می‌شه یا می‌شه همونجا اون پوش و وارد گیت‌هابش کرد. اونجا مدل‌ها رو بتونم انتخاب بکنم که این قبل از اینکه حالا مثلاً پوش بشه یا واحد محلی اجرایی بشه و دیگه ممکنه بعداً دیر بشه." این درخواست بر ایجاد یک نقطه ورود (entry point) متمرکز است که امکان انتخاب مدل‌ها را فراهم می‌کند، نه پیاده‌سازی کامل منطق بازبینی یا رابط کاربری پیچیده انتخاب مدل. این نقطه اتصال باید در جایی قرار گیرد که پروژه با "گیت‌هاب" تعامل دارد یا برای "واحد محلی اجرایی" آماده می‌شود.

با توجه به ساختار پروژه، این نقطه اتصال می‌تواند یک API جدید در `backend/app/api/routes/projects.py` باشد که توسط `frontend` قبل از عملیات نهایی پوش یا اجرا فراخوانی می‌شود. این API باید با `backend/app/services/project_service.py` برای مدیریت منطق و با `backend/app/models/project.py` برای ذخیره‌سازی تنظیمات پروژه و `backend/app/models/ai_profile.py` برای اعتبارسنجی مدل‌های انتخابی تعامل داشته باشد. کلیدواژه‌های اصلی شامل: گیت‌هاب, پوش, واحد محلی اجرایی, پروژه, صفحه گیت‌هاب, مدل‌ها رو بتونم انتخاب بکنم.

بخش مربوط از درخواست اصلی کاربر: «با ایده‌هایی که داشتم که باعث ساخته شدن اون فایل‌ها شده رو بتونم مدل یا مدل‌ها رو انتخاب کنم که کامل بررسی کنم و مثلاً ایده رو مثلاً یا مشکل مثلاً یا هرگونه مدل هم بتونه یه نقشی بگیره و از یه جهتی بررسی بکنن سوالات رو.»

کلیدواژه‌های اصلی از درخواست کاربر شامل: «ایده‌ها, فایل‌ها, ساخته شدن, بررسی» است.

برای پیاده‌سازی این قابلیت، نیاز به تعریف یک مدل داده‌ای جدید برای ذخیره ایده‌ها، ایجاد سرویس‌های مربوط به CRUD (Create, Read, Update, Delete) این ایده‌ها، و افزودن endpointهای API برای تعامل با آن‌ها داریم. سپس، سرویس‌های تحلیل و بازبینی مبتنی بر AI (مانند `backend/app/services/oversight_deep_scan_service.py` یا `backend/app/services/inspector_agent.py`) باید به‌روزرسانی شوند تا بتوانند این ایده‌ها را به عنوان بخشی از context ورودی خود دریافت و پردازش کنند. این مکانیزم باید به `backend/app/services/ai_manager.py` اجازه دهد تا مدل‌های مناسب را برای بررسی این ایده‌ها انتخاب کند و آن‌ها را به همراه محتوای فایل‌ها برای تحلیل جامع به مدل‌های AI ارسال کند.

## ⚠️ ریسک‌ها و موارد احتیاط
1.  **Database Schema Migration:** معرفی مدل `ProjectIdea` جدید نیازمند یک migration دیتابیس است که باید با دقت برنامه‌ریزی و اجرا شود تا از از دست رفتن داده‌ها یا اختلال در سرویس جلوگیری شود.
2.  **AI Context Window Limits:** ارسال اطلاعات اضافی 'ایده' به مدل‌های AI (به‌ویژه از طریق `backend/app/services/oversight_deep_scan_service.py`) می‌تواند به سرعت از محدودیت‌های context window مدل فراتر رود. این امر نیازمند استراتژی‌هایی مانند خلاصه‌سازی، فیلتر کردن هوشمند یا انتخاب گزینشی ایده‌ها است.

4.  **Data Consistency and Redundancy:** اگر 'ایده‌ها' شامل لینک به اسناد خارجی باشند، اطمینان از اعتبار و به‌روز بودن این لینک‌ها و هماهنگی آن‌ها با فراداده‌های ذخیره شده، یک چالش خواهد بود.

**Key terms:** صفحه گیت‌هاب, مدل‌ها, انتخاب بکنم, اختیاری, انتخاب کنم

**بخش مربوط از متن کاربر:**
```
می‌خوام توی صفحه گیت‌هاب، اونجایی که پروژه حالا به صورت محلی ساخته می‌شه یا می‌شه همونجا اون پوش و وارد گیت‌هابش کرد. اونجا مدل‌ها رو بتونم انتخاب بکنم که ... یه مرحله دیگه هم باشه که اونجا بتونه حالا اختیاری باشه و که مدل یا مدل‌هایی رو اونجا انتخاب کنم.
```

## 🌐 نقشهٔ وابستگی‌ها
این قابلیت به شدت به مدل `Project` در `backend/app/models/project.py` وابسته است که نیاز به یک فیلد جدید برای ذخیره تنظیمات مدل‌های بازبینی دارد. سرویس `backend/app/services/project_service.py` مسئول تعامل با این مدل خواهد بود. برای لیست کردن مدل‌های AI موجود، `backend/app/services/ai_manager.py` فراخوانی خواهد شد. در نهایت، سرویس‌های اصلی بازبینی مانند `backend/app/services/oversight_service.py` یا `backend/app/services/inspector_agent.py` باید تنظیمات ذخیره شده را از `Project` بخوانند و از آن‌ها برای اجرای فرآیند بازبینی استفاده کنند. در فرانت‌اند، یک کامپوننت جدید در `frontend/` به APIهای بک‌اند برای دریافت لیست مدل‌ها و ذخیره تنظیمات وابسته خواهد بود.

## 🔍 Context و وضعیت فعلی
کاربر درخواست طراحی و پیاده‌سازی یک رابط کاربری (UI) جدید را دارد که به کاربران امکان می‌دهد مدل‌های بازبینی (review models) را برای پروژه‌های گیت‌هاب یا محلی انتخاب و پیکربندی کنند. این رابط کاربری باید در "صفحه گیت‌هاب" یا در جریان ایجاد/وارد کردن "پروژه محلی" قرار گیرد. هدف این است که کاربران بتوانند "مدل‌ها"ی موجود را مرور کرده، "یک یا چند مورد را برای بازبینی انتخاب کنند"، "پارامترهای آن‌ها را (در صورت لزوم) پیکربندی کنند" و "فرآیند بازبینی را آغاز کنند". تأکید شده است که این مرحله "اختیاری" است و صرفاً بر اجزای بصری و تعاملی برای "انتخاب مدل" تمرکز دارد.

این قابلیت به کاربران اجازه می‌دهد تا کنترل بیشتری بر فرآیند بازبینی کد و پروژه داشته باشند، به ویژه در سناریوهایی که پروژه از گیت‌هاب وارد شده یا به صورت محلی مدیریت می‌شود. با توجه به ساختار پروژه، این ویژگی نیاز به تغییرات در هر دو بخش فرانت‌اند (برای UI) و بک‌اند (برای API و ذخیره‌سازی تنظیمات) دارد.
در بک‌اند، فایل‌هایی مانند `backend/app/api/routes/github_import.py` و `backend/app/services/github_import.py` ممکن است نقطه ورود برای این قابلیت باشند، در حالی که `backend/app/api/routes/models.py` و `backend/app/services/ai_manager.py` برای لیست کردن مدل‌ها و `backend/app/models/project.py` برای ذخیره تنظیمات مدل در پروژه مرتبط هستند.
در فرانت‌اند، نیاز به ایجاد یک کامپوننت جدید برای نمایش لیست مدل‌ها، چک‌باکس‌های انتخاب، فیلدهای ورودی پارامتر و دکمه‌های اقدام است.

from fastapi import APIRouter, Depends, HTTPException
from sqlalchemy.ext.asyncio import AsyncSession
from backend.app.core.database import get_db
from backend.app.services.project_service import ProjectService
from backend.app.schemas.project import ProjectReviewModelConfig  # New Pydantic schema

router = APIRouter()

@router.post("/{project_id}/review-model-config")
async def configure_project_review_models(
    project_id: int,
    config: ProjectReviewModelConfig,
    db: AsyncSession = Depends(get_db)
):
    project_service = ProjectService(db)
    updated_project = await project_service.update_project_review_config(project_id, config.dict())
    if not updated_project:
        raise HTTPException(status_code=404, detail="Project not found")
    return {"message": "Review model configuration updated successfully"}

- `backend/app/services/deploy_service.py` — کاربر به صراحت به «پوش/اجرا» پس از تأیید اشاره می‌کند. این سرویس مرحله منطقی بعدی است که باید توسط نقطه پایانی `POST /api/oversight/projects/{project_id}/review_report/action` پس از تأیید موفقیت‌آمیز فعال شود.
- `backend/app/models/project.py` — مدل پروژه باید وضعیت کلی تأیید (مثلاً `review_status: str = "PENDING"`) را ذخیره کند که توسط اقدام تأیید به‌روزرسانی می‌شود.
- `frontend/package.json` — این فایل با افزودن هرگونه کتابخانه UI جدید به‌روزرسانی خواهد شد و اسکریپت‌های ساخت/اجرای فرانت‌اند مرتبط خواهند بود.

## 🌐 نقشهٔ وابستگی‌ها

## 🔍 Context و وضعیت فعلی
کاربر درخواست طراحی و پیاده‌سازی یک رابط کاربری (UI) برای نمایش نتایج بازبینی و تأیید جامع را دارد. این UI برای ارائه گزارش بازبینی تولید شده توسط مدل‌ها (که در متن کاربر به عنوان «مرحله ۷» اشاره شده) حیاتی است. گزارش باید به وضوح مدل‌های اجرا شده، یافته‌های آن‌ها، هرگونه مشکل شناسایی شده، پیشنهادات و وضعیت تأیید نهایی تجمیع شده را نمایش دهد. یک نیاز کلیدی، فراهم کردن اقدامات واضح برای کاربر است تا بر اساس بازبینی، پروژه را «تأیید» (Approve) یا «ادامه» (Continue) دهد. این قابلیت به کاربران امکان می‌دهد تا «با خیال راحت» به مرحله بعدی (مانند «پوش/اجرا» یا Deploy/Execute) بروند. کاربر همچنین تأکید می‌کند که این فرآیند شامل «بررسی همه فایل‌ها»، «ایده‌ها»، «تحلیل و ساخت» برای ارائه «تأییدیه» است. این نشان‌دهنده یک مرحله حیاتی «انسان در حلقه» (human-in-the-loop) در چرخه عمر پروژه است که در آن تحلیل خودکار برای تأیید نهایی انسانی ارائه می‌شود. سرویس‌های بک‌اند مانند `backend/app/services/oversight_service.py`، `backend/app/services/oversight_verifier.py` و `backend/app/services/oversight_deep_scan_service.py` احتمالاً مسئول تولید این گزارش‌ها هستند، در حالی که `backend/app/api/routes/oversight.py` نقاط پایانی API لازم را برای این منظور فراهم خواهد کرد. فرانت‌اند این APIها را برای رندر کردن گزارش تعاملی مصرف خواهد کرد.

import { useRouter } from 'next/router';
import { useEffect, useState } from 'react';
import axios from 'axios';
import { ComprehensiveReviewReport } from '../../../types/oversight'; // New type definition

interface ReviewReport {
    models_executed: string[];
    findings: any[];
    issues_identified: any[];
    suggestions: string[];
    final_verification_status: string;
}

const ProjectReviewPage = () => {
    const router = useRouter();
    const { projectId } = router.query;
    const [report, setReport] = useState<ReviewReport | null>(null);
    const [loading, setLoading] = useState(true);
    const [error, setError] = useState<string | null>(null);

    useEffect(() => {
        if (projectId) {
            const fetchReport = async () => {
                try {
                    const response = await axios.get(`/api/oversight/projects/${projectId}/review_report`);
                    setReport(response.data);
                } catch (err) {
                    setError("Failed to load review report.");
                    console.error(err);
                } finally {
                    setLoading(false);
                }
            };
            fetchReport();
        }
    }, [projectId]);

    const handleApprove = async () => {
        try {
            await axios.post(`/api/oversight/projects/${projectId}/review_report/action`, { action_type: "APPROVE" });
            alert("Project approved successfully!");
            router.push(`/projects/${projectId}/deploy`); // Navigate to next stage
        } catch (err) {
            alert("Failed to approve project.");
        }
    };

    const handleContinue = async () => {
        try {
            await axios.post(`/api/oversight/projects/${projectId}/review_report/action`, { action_type: "CONTINUE" });
            alert("Project review continued.");
            router.push(`/projects/${projectId}/next-step`); // Navigate to another stage
        } catch (err) {
            alert("Failed to continue review.");
        }
    };

    if (loading) return <div>Loading review report...</div>;
    if (error) return <div>Error: {error}</div>;
    if (!report) return <div>No review report available.</div>;

    return (
        <div>
            <h1>Comprehensive Review Report for Project {projectId}</h1>
            <p>Status: {report.final_verification_status}</p>
            <h2>Models Executed:</h2>
            <ul>{report.models_executed.map((model, i) => <li key={i}>{model}</li>)}</ul>
            <h2>Findings:</h2>
            {/* Render findings, issues, suggestions */}
            <button onClick={handleApprove}>Approve</button>
            <button onClick={handleContinue}>Continue</button>
        </div>
    );
};

export default ProjectReviewPage;
```

**Excerpt:**
```
می‌خوام توی صفحه گیت‌هاب، اونجایی که پروژه حالا به صورت محلی ساخته می‌شه یا می‌شه همونجا اون پوش و وارد گیت‌هابش کرد. اونجا مدل‌ها رو بتونم انتخاب بکنم که این قبل از اینکه حالا مثلاً پوش بشه یا واحد محلی اجرایی بشه و دیگه ممکنه بعداً دیر بشه.
