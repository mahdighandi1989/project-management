# 🧬 Bug C3 — Post-Verify Intelligent Task Consolidation Engine (نسخهٔ اصلاح‌شده)

## 🎯 هدف کلی

بعد از پایان هر **Bulk Verify** (چه `fast` چه `deep`)، تسک‌هایی که به وضعیت done نرسیده‌اند (`not_done`، `partial`، `needs_clarification`، `regressed`) به‌جای ماندن به‌صورت ۲۰۵ تسک پراکنده، **به‌صورت هوشمند بر اساس ماهیت و شباهت** در چند تسک بزرگ‌تر تجمیع شوند.

---

## 📐 محل اجرا و trigger

- نقطه: **انتهای `_run_bulk_verify`** در `backend/app/api/routes/oversight.py` — بعد از `await _aio.gather(...)` و قبل از Telegram notification.
- شرط‌های اجرا:
  - فقط روی همان `watched_id`
  - فقط روی تسک‌هایی که `source == "auto_scan"`
  - فقط تسک‌های غیر done و غیر archived
  - حداقل ۵ تسک باقی‌مانده
  - **روی fast و deep یکسان** (mode فرق نمی‌کند)

## 🧠 الگوریتم clustering — **سه مرحله‌ای** (اصلاح بند ۸)

### 🔄 مرحلهٔ ۱ — Live identification + پیش‌دسته‌بندی **در حین verify**

داخل حلقهٔ `_verify_one` که الان کار می‌کند:

#### الف) جمع‌آوری fingerprint غنی (no AI)
برای هر تسکی که verify نهایی‌اش done نیست، fingerprint ذخیره می‌شود در `_BULK_VERIFY_STATE["consolidation_candidates"]`:
- `task_id`, `title`, `idea_prompt`, `priority`, `tags`, `type`
- `acceptance_criteria` کامل
- `task_steps` (با وضعیت per-step verify — برای partialها حیاتی)
- `files_touched` از evidence و commit diff
- `verification_status`, `verification_report.last_status`
- `remaining_parts` و `completed_parts` (تفکیک شده برای partialها)
- `creation_context`

#### ب) Rolling pre-cluster در حین verify (**جدید**، بند ۸)
هر چند تسک یک‌بار (مثلاً هر ۱۰ تسک تکمیل‌شده)، یک تابع mechanical لاجیک **بدون AI** اجرا می‌شود که کاندیدهای تجمعی فعلی را گروه‌بندی اولیه می‌کند:
- **same-file group**: Jaccard ≥ 0.5 روی `files_touched`
- **same-tag group**: حداقل یک tag مشترک با کاندیدای دیگر
- **title-trigram group**: شباهت کسینوسی > 0.7
- نتیجه در `_BULK_VERIFY_STATE["live_preclusters"]` ذخیره می‌شود
- UI می‌تواند به‌صورت زنده نشان دهد: "🧬 پیش‌بینی: حدود N cluster"

این فقط hint برای مرحلهٔ ۳ است — نه تصمیم نهایی.

### 🧮 مرحلهٔ ۲ — AI Clustering در پایان verify

پس از `_aio.gather` و قبل از Telegram:

- مدل: `DEFAULT_EXTRACTION_MODEL_ID` (DeepSeek-Chat)
- ورودی: کاندیدها + live_preclusters به‌عنوان hint + meta هر تسک
- اگر کاندیدها > ۴۰، batch های ۴۰تایی با حفظ context قبلی
- خروجی JSON:
  ```json
  {
    "clusters": [{
      "cluster_id": "...",
      "theme": "...",
      "rationale": "...",
      "task_ids": [...],
      "suggested_super_title": "...",
      "priority": "critical|high|medium|low",
      "type": "frontend|backend|fullstack|infra",
      "estimated_difficulty": "small|medium|large",
      "mixed_partial_and_not_done": true|false
    }],
    "unclustered_task_ids": [...]
  }
  ```

**🔑 صراحت در system prompt (اصلاح بند ۱۰+۱۱):**
> "تسک‌های partial **حتماً باید** در همان clusterهایی قرار بگیرند که با not_done ها هم‌موضوع هستند. **هرگز** برای partialها cluster مجزا نساز. اگر یک partial کنار یک not_done قرار می‌گیرد، در super-task به‌صورت 'این بخش‌ها قبلاً انجام شده، فقط remaining باقی مانده' متمایز شود."

### ✅ مرحلهٔ ۳ — Validation
- task_id ناشناخته drop
- task_id تکراری بین cluster ها → اولی برنده
- cluster های تک‌تسکی drop → به unclustered
- اگر AI کاملاً ناموفق → fallback به live_preclusters
- اگر آن هم چیزی نداد → consolidation skip با error log (verify summary سالم می‌ماند)

### `task_steps` تولید (**اصلاح بند ۱۳ — مهم‌ترین تغییر**)

**به‌جای** یک AI call ساده برای merging مراحل، **همان pipeline موجود برای ایده‌های خام بلند** استفاده می‌شود:

> `idea_prompt` تلفیقی (که طول آن می‌تواند ۵۰۰۰+ کلمه باشد) به تابع موجود `_ai_plan_steps_from_idea` که برای bug 30 v2 بهینه شد (۳-tier strategy: monolithic → retry → **programmatic chunking با parallel AI calls per section**) ارسال می‌شود.

این یعنی:
- ساختار super-task مثل یک "ایدهٔ خام بلند جدید" رفتار می‌کند
- اگر AI نتوانست monolithically همهٔ steps را تولید کند، خودکار به chunking می‌رود
- خروجی همان فرمت `task_steps` استاندارد است — کاملاً سازگار با بقیهٔ سیستم

اضافه: مراحل common بین source taskها (مثلاً "تست نوشتن" که در ۵ تسک مشترک بود) **یک‌بار** در super-task ظاهر می‌شوند، با نشانهٔ `(merged from N tasks)`.

مراحلی که در همهٔ source ها done بودند → در super-task علامت `pre_done: true` می‌خورند (نه پاک می‌شوند، نه تکرار).

### `acceptance_criteria`
- union همهٔ AC ها از source ها
- dedup روی text lowercase (کاراکترهای فاصله، نقطه، ... نرمالایز)
- هر AC ساختار کامل (verify_method, verify_plan, evidence_locations) را حفظ می‌کند
- AC هایی که قبلاً در reportها به‌عنوان "passed" verify شده‌اند، با `pre_passed: true` علامت می‌خورند

### Intelligent Checklist (شرطی) — **اصلاح بند ۵**

شرط دقیق:
- `watched.intelligent_features_enabled == True` (نام دقیق فلگ هنگام پیاده‌سازی از روی schema تأیید می‌شود — هر چه هست، اولین کار query کردن watched object برای flag مرتبط)
- **و** سیستم تشخیص دهد cluster `estimated_difficulty in ("medium", "large")`
- **و** تعداد source taskها ≥ ۳ (کمتر از این نیازی به checklist نیست)

اگر هر سه شرط برقرار شد:
- یک AI call سوم با ورودی `idea_prompt` تلفیقی + AC ها
- خروجی: checklist با ۵-۱۵ آیتم، هر آیتم با priority
- ذخیره در `task.intelligent_checklist` (یا هر فیلدی که موجود است)

اگر شرط برقرار نبود → skip بدون error، super-task بدون checklist ساخته می‌شود.

### سایر فیلدها
- `priority`: AI cluster priority
- `type`: AI cluster type
- `tags`: union همهٔ tagها + `["consolidated", "post_verify_merge"]`
- `source`: `"auto_consolidation"` (نوع جدید)
- `merged_from`: لیست task_idهای source
- `consolidation_meta`: `{cluster_theme, rationale, created_during_bulk_verify_at, mode, ai_calls_used}`
- `verification_status`: `"pending"`

## ⚠️ Edge cases و safeguards

1. AI failure → fallback به live_preclusters → اگر آن هم نشد، skip بدون شکست verify
2. snapshot کامل source ها در super-task برای undo
3. تسک‌های user دست‌نخورده (source_filter رعایت می‌شود)
4. `_CONSOLIDATION_LOCK` module-level — جلوگیری از concurrent
5. کاندیدها > ۲۰۰ → top-200 by priority؛ بقیه نوبت بعد
6. Idempotency: تسک با tag `consolidated` یا `merged` re-consolidate نمی‌شود
7. **partial-only cluster ها هم مجازند** اگر AI تشخیص دهد ۲ partial با ماهیت یکسان — اما برای تأکید بند ۱۱، system prompt اولویت می‌دهد به cluster mixed (partial + not_done)

---

## 🧪 معیار موفقیت (اصلاح)

1. روی ۲۰۵ تسک کاربر → ≥ ۱۵ cluster و کاهش به ۲۰-۳۰ super-task
2. هیچ تسک done نباید آرشیو شود (test invariant)
3. **مجموع طول `idea_prompt` همهٔ source ها ≤ طول `super_task.idea_prompt`** (allow کوچک overhead برای headers — این invariant جلوی خلاصه‌سازی را می‌گیرد)
4. تسک‌های partial حداقل ۸۰٪ در cluster mixed (نه pure-partial)
5. مراحل done شده در source → در super-task `pre_done` (نه remaining)
6. UI super-taskها را با badge `🧬 consolidated` متمایز کند
7. undo با یک کلیک کار کند

## 🧠 الگوریتم clustering — **سه مرحله‌ای** (اصلاح بند ۸)

### 🔄 مرحلهٔ ۱ — Live identification + پیش‌دسته‌بندی **در حین verify**

داخل حلقهٔ `_verify_one` که الان کار می‌کند:

#### الف) جمع‌آوری fingerprint غنی (no AI)
برای هر تسکی که verify نهایی‌اش done نیست، fingerprint ذخیره می‌شود در `_BULK_VERIFY_STATE["consolidation_candidates"]`:
- `task_id`, `title`, `idea_prompt`, `priority`, `tags`, `type`
- `acceptance_criteria` کامل
- `task_steps` (با وضعیت per-step verify — برای partialها حیاتی)
- `files_touched` از evidence و commit diff
- `verification_status`, `verification_report.last_status`
- `remaining_parts` و `completed_parts` (تفکیک شده برای partialها)
- `creation_context`

#### ب) Rolling pre-cluster در حین verify (**جدید**، بند ۸)
هر چند تسک یک‌بار (مثلاً هر ۱۰ تسک تکمیل‌شده)، یک تابع mechanical لاجیک **بدون AI** اجرا می‌شود که کاندیدهای تجمعی فعلی را گروه‌بندی اولیه می‌کند:
- **same-file group**: Jaccard ≥ 0.5 روی `files_touched`
- **same-tag group**: حداقل یک tag مشترک با کاندیدای دیگر
- **title-trigram group**: شباهت کسینوسی > 0.7
- نتیجه در `_BULK_VERIFY_STATE["live_preclusters"]` ذخیره می‌شود
- UI می‌تواند به‌صورت زنده نشان دهد: "🧬 پیش‌بینی: حدود N cluster"

این فقط hint برای مرحلهٔ ۳ است — نه تصمیم نهایی.

### 🧮 مرحلهٔ ۲ — AI Clustering در پایان verify

پس از `_aio.gather` و قبل از Telegram:

- مدل: `DEFAULT_EXTRACTION_MODEL_ID` (DeepSeek-Chat)
- ورودی: کاندیدها + live_preclusters به‌عنوان hint + meta هر تسک
- اگر کاندیدها > ۴۰، batch های ۴۰تایی با حفظ context قبلی
- خروجی JSON:
  ```json
  {
    "clusters": [{
      "cluster_id": "...",
      "theme": "...",
      "rationale": "...",
      "task_ids": [...],
      "suggested_super_title": "...",
      "priority": "critical|high|medium|low",
      "type": "frontend|backend|fullstack|infra",
      "estimated_difficulty": "small|medium|large",
      "mixed_partial_and_not_done": true|false
    }],
    "unclustered_task_ids": [...]
  }
  ```

**🔑 صراحت در system prompt (اصلاح بند ۱۰+۱۱):**
> "تسک‌های partial **حتماً باید** در همان clusterهایی قرار بگیرند که با not_done ها هم‌موضوع هستند. **هرگز** برای partialها cluster مجزا نساز. اگر یک partial کنار یک not_done قرار می‌گیرد، در super-task به‌صورت 'این بخش‌ها قبلاً انجام شده، فقط remaining باقی مانده' متمایز شود."

### ✅ مرحلهٔ ۳ — Validation
- task_id ناشناخته drop
- task_id تکراری بین cluster ها → اولی برنده
- cluster های تک‌تسکی drop → به unclustered
- اگر AI کاملاً ناموفق → fallback به live_preclusters
- اگر آن هم چیزی نداد → consolidation skip با error log (verify summary سالم می‌ماند)

### `task_steps` تولید (**اصلاح بند ۱۳ — مهم‌ترین تغییر**)

**به‌جای** یک AI call ساده برای merging مراحل، **همان pipeline موجود برای ایده‌های خام بلند** استفاده می‌شود:

> `idea_prompt` تلفیقی (که طول آن می‌تواند ۵۰۰۰+ کلمه باشد) به تابع موجود `_ai_plan_steps_from_idea` که برای bug 30 v2 بهینه شد (۳-tier strategy: monolithic → retry → **programmatic chunking با parallel AI calls per section**) ارسال می‌شود.

این یعنی:
- ساختار super-task مثل یک "ایدهٔ خام بلند جدید" رفتار می‌کند
- اگر AI نتوانست monolithically همهٔ steps را تولید کند، خودکار به chunking می‌رود
- خروجی همان فرمت `task_steps` استاندارد است — کاملاً سازگار با بقیهٔ سیستم

اضافه: مراحل common بین source taskها (مثلاً "تست نوشتن" که در ۵ تسک مشترک بود) **یک‌بار** در super-task ظاهر می‌شوند، با نشانهٔ `(merged from N tasks)`.

مراحلی که در همهٔ source ها done بودند → در super-task علامت `pre_done: true` می‌خورند (نه پاک می‌شوند، نه تکرار).

### `acceptance_criteria`
- union همهٔ AC ها از source ها
- dedup روی text lowercase (کاراکترهای فاصله، نقطه، ... نرمالایز)
- هر AC ساختار کامل (verify_method, verify_plan, evidence_locations) را حفظ می‌کند
- AC هایی که قبلاً در reportها به‌عنوان "passed" verify شده‌اند، با `pre_passed: true` علامت می‌خورند

### Intelligent Checklist (شرطی) — **اصلاح بند ۵**

شرط دقیق:
- `watched.intelligent_features_enabled == True` (نام دقیق فلگ هنگام پیاده‌سازی از روی schema تأیید می‌شود — هر چه هست، اولین کار query کردن watched object برای flag مرتبط)
- **و** سیستم تشخیص دهد cluster `estimated_difficulty in ("medium", "large")`
- **و** تعداد source taskها ≥ ۳ (کمتر از این نیازی به checklist نیست)

اگر هر سه شرط برقرار شد:
- یک AI call سوم با ورودی `idea_prompt` تلفیقی + AC ها
- خروجی: checklist با ۵-۱۵ آیتم، هر آیتم با priority
- ذخیره در `task.intelligent_checklist` (یا هر فیلدی که موجود است)

اگر شرط برقرار نبود → skip بدون error، super-task بدون checklist ساخته می‌شود.

### سایر فیلدها
- `priority`: AI cluster priority
- `type`: AI cluster type
- `tags`: union همهٔ tagها + `["consolidated", "post_verify_merge"]`
- `source`: `"auto_consolidation"` (نوع جدید)
- `merged_from`: لیست task_idهای source
- `consolidation_meta`: `{cluster_theme, rationale, created_during_bulk_verify_at, mode, ai_calls_used}`
- `verification_status`: `"pending"`

## ⚠️ Edge cases و safeguards

1. AI failure → fallback به live_preclusters → اگر آن هم نشد، skip بدون شکست verify
2. snapshot کامل source ها در super-task برای undo
3. تسک‌های user دست‌نخورده (source_filter رعایت می‌شود)
4. `_CONSOLIDATION_LOCK` module-level — جلوگیری از concurrent
5. کاندیدها > ۲۰۰ → top-200 by priority؛ بقیه نوبت بعد
6. Idempotency: تسک با tag `consolidated` یا `merged` re-consolidate نمی‌شود
7. **partial-only cluster ها هم مجازند** اگر AI تشخیص دهد ۲ partial با ماهیت یکسان — اما برای تأکید بند ۱۱، system prompt اولویت می‌دهد به cluster mixed (partial + not_done)

---

## 🧪 معیار موفقیت (اصلاح)

1. روی ۲۰۵ تسک کاربر → ≥ ۱۵ cluster و کاهش به ۲۰-۳۰ super-task
2. هیچ تسک done نباید آرشیو شود (test invariant)
3. **مجموع طول `idea_prompt` همهٔ source ها ≤ طول `super_task.idea_prompt`** (allow کوچک overhead برای headers — این invariant جلوی خلاصه‌سازی را می‌گیرد)
4. تسک‌های partial حداقل ۸۰٪ در cluster mixed (نه pure-partial)
5. مراحل done شده در source → در super-task `pre_done` (نه remaining)
6. UI super-taskها را با badge `🧬 consolidated` متمایز کند
7. undo با یک کلیک کار کند

1. **AI failure → fallback به live_preclusters**: اگر AI در خوشه‌بندی/ادغام تسک‌ها شکست خورد (exception یا timeout)، به جای توقف کل فرآیند، از `live_preclusters` (خوشه‌های از پیش محاسبه‌شده بر اساس شباهت متنی/ساختاری) استفاده شود. اگر `live_preclusters` هم خالی/ناموفق بود، تسک‌ها بدون ادغام (skip) عبور کنند و verify شکست نخورد.
