# 🎯 هدف کلی (6 مرحله)

## Raw Idea

🎯 هدف کلی
تقسیم Bulk Verify به دو دکمهٔ مستقل بدون selector dialog میانی:

🔬 Bulk Verify (سبز) → فقط mode deep
▶ بررسی فوری (آبی) → فقط mode fast (جایگزین کامل کار قبلی)
هر دو دکمه از سه تابع یکتای مشترک عبور می‌کنند:

یک تابع frontend واحد: startBulkVerify(mode: 'fast'|'deep')
یک تابع backend واحد: _run_bulk_verify(...mode=...)
یک تابع consolidation واحد: consolidate_remaining_tasks(...) (از C3)
🔒 INVARIANT حیاتی (بند ۱۲ + ۱۳):

در هیچ‌یک از این سه تابع، هیچ شاخهٔ if/else بر اساس mode نباید در منطق consolidation وجود داشته باشد. mode فقط تعیین می‌کند که verify چقدر عمیق اجرا شود (که خارج از scope consolidation است). دقت، الگوریتم، AI calls، prompt ها، validation، snapshot، و undo — همه برای هر دو حالت 100% یکسان است.

اگر در review کد یک نفر یک if mode == "fast" در task_consolidation_service.py ببیند → fail. اگر در startBulkVerify یک if mode === 'fast' در منطق پسا-verify ببیند → fail.

🔍 مرحلهٔ ۰ — Audit (read-only، report-back قبل از تغییر) — بند ۲
قبل از هر تغییر، این موارد را بررسی می‌کنم و گزارش به شما برمی‌گردانم، هیچ کدی تغییر نمی‌دهم تا تأیید کنید:

onRunNow در frontend — کجا تعریف شده، چه می‌کند، چه endpoint backend صدا می‌زند
callsites دیگر onRunNow — فقط همین دکمه استفاده می‌کند یا جای دیگر هم؟
endpoint backend مرتبط — آیا scheduler/autonomy/auto-loop به آن وابسته‌اند؟
آیا قبلاً بود برای verify بود یا execution؟ — فرضیهٔ شما را با مدرک تأیید/رد می‌کنم
گزارش این چهار مورد قبل از هر تغییر در یک پیام کوتاه به شما داده می‌شود. این جزء جدایی‌ناپذیر pipeline است، نه optional.

سپس بسته به نتیجه:

اگر onRunNow کاملاً isolated بود → حذف کامل با prop chain تمیز
اگر scheduler به آن وابسته بود → فقط dis-connect دکمه، تابع backend زنده
اگر یکی از دکمه‌های دیگر هم استفاده می‌کرد → keep و فقط onClick این دکمه عوض
📐 تغییرات Backend
_run_bulk_verify
هیچ تغییر نمی‌کند. signature فعلی کامل است:

async def _run_bulk_verify(
    watched_id, priority_filter, auto_archive_done, concurrency, *,
    source_filter="auto_scan", mode="deep", auto_consolidate=True,
) -> None
Invariant: هیچ خط جدیدی به این تابع اضافه نمی‌شود که بر اساس mode رفتار consolidation را عوض کند.

endpoint POST /watched/{id}/bulk-verify
هیچ تغییر نمی‌کند. همان parameters. فقط frontend دو call مختلف می‌زند.

endpoint قدیمی onRunNow (هرچه باشد)
بر اساس نتیجهٔ audit مرحلهٔ ۰:

اگر unused → حذف کامل
اگر scheduler-used → نگه می‌دارم با هشدار در commit message
