# [منطق] عدم coherence بین model و service در مدیریت session_id

## Raw Idea

## 📋 شرح ناسازگاری
در pipeline `auth` یک ناسازگاری منطقی پیدا شد:

مدل inspector_session شامل session_id به عنوان کلید اصلی است، اما سرویس oversight_upload_session از session_id برای resume استفاده می‌کند. مشخص نیست که آیا این session_idها یکسان هستند یا متفاوت. همچنین مدل inspector_session فیلدهایی مثل project_id دارد که در سرویس آپلود وجود ندارد.

## 💥 پیامد (impact)
اگر session_idها overlap داشته باشند، ممکن است داده‌های سشن آپلود با داده‌های inspector session اشتباه گرفته شوند. این باعث corruption داده یا رفتار غیرمنتظره می‌شود.

## 🛠 پیشنهاد رفع اولیه
یک namespace یا prefix برای session_idهای آپلود در نظر بگیرید (مثلاً 'upload_' + uuid). یا از یک جدول/مدل جداگانه برای upload sessions استفاده کنید تا با inspector sessions تداخل نداشته باشد.

## 🤔 چرا مهم است
coherence issue یعنی دو بخش کد فرض‌های ناسازگار دارند — معمولاً نشانه‌ی refactor ناتمام یا feature flag rot است. این کلاس bug ها در test معمولی پیدا نمی‌شوند چون unit test ها در silo اجرا می‌شوند.
