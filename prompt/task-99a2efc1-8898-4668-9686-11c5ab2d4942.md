# [منطق] احتمال info leak در permission checking

## Raw Idea

## 📋 شرح ناسازگاری
در pipeline `auth` یک ناسازگاری منطقی پیدا شد:

اگر permission checking وجود داشته باشد، اما خطاهای دسترسی اطلاعات اضافی فاش کنند (مثلاً 'شما دسترسی به پروژه X را ندارید' که وجود پروژه X را تأیید می‌کند)، این یک info leak است. در توضیحات فعلی هیچ اشاره‌ای به نحوه مدیریت خطاهای دسترسی نشده است.

## 💥 پیامد (impact)
مهاجم می‌تواند با استفاده از خطاهای دسترسی، اطلاعاتی درباره موجودیت‌های سیستم (مثلاً وجود پروژه‌ها یا کاربران) به دست آورد.

## 🛠 پیشنهاد رفع اولیه
خطاهای دسترسی را generic کنید. مثلاً همیشه 'دسترسی غیرمجاز' برگردانید بدون ذکر جزئیات. از exception handlers استاندارد استفاده کنید.

## 🤔 چرا مهم است
coherence issue یعنی دو بخش کد فرض‌های ناسازگار دارند — معمولاً نشانه‌ی refactor ناتمام یا feature flag rot است. این کلاس bug ها در test معمولی پیدا نمی‌شوند چون unit test ها در silo اجرا می‌شوند.
