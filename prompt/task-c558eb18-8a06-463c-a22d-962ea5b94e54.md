# [منطق] عدم وجود caching strategy برای انتخاب مدل در ai_manager

## Raw Idea

## 📋 شرح ناسازگاری
در pipeline `ai_llm` یک ناسازگاری منطقی پیدا شد:

ai_manager وظیفه انتخاب هوشمند مدل و load balancing را دارد، اما هیچ مکانیزم caching برای نتایج انتخاب مدل (مثلاً بر اساس task_type + capabilities) ذکر نشده است. هر بار که درخواست می‌آید، ممکن است محاسبات انتخاب مدل تکرار شود.

## 💥 پیامد (impact)
در سناریوهای high-load، انتخاب مکرر مدل باعث افزایش latency و مصرف منابع (API calls به model registry, database queries) می‌شود. این می‌تواند bottleneck ایجاد کند.

## 🛠 پیشنهاد رفع اولیه
یک cache لایه‌ای (مثلاً Redis یا in-memory با TTL) برای نتایج انتخاب مدل اضافه کنید. کلید cache ترکیبی از task_type + capabilities + active providers باشد. در صورت تغییر تنظیمات، cache را invalidate کنید.

## 🤔 چرا مهم است
coherence issue یعنی دو بخش کد فرض‌های ناسازگار دارند — معمولاً نشانه‌ی refactor ناتمام یا feature flag rot است. این کلاس bug ها در test معمولی پیدا نمی‌شوند چون unit test ها در silo اجرا می‌شوند.
