# Analysis page uses hardcoded fallback data for model profiles instead of real backend data

## Raw Idea

The frontend `analysis/page.tsx` and `model-profiles/page.tsx` pages contain extensive hardcoded default profile data (e.g., `defaultProfiles`, `defaultLeaderboard`) that are used when the backend API returns empty or fails. This means the UI always shows fake data unless a real analysis has been run. The backend `model_profiles.py` route also has a fallback mechanism (`_get_default_profile_list`) that returns hardcoded data. This creates a misleading user experience where the system appears to have model rankings and scores that are not based on actual performance.

## 🌐 نقشهٔ وابستگی‌ها
This affects the entire model profiling feature. The backend `model_profiles.py` route is imported by `main.py` and serves the `/api/models/profiles` endpoint. The frontend pages consume this endpoint. The fallback data is used in 3 separate locations.

## 🔍 Context و وضعیت فعلی
The frontend `analysis/page.tsx` and `model-profiles/page.tsx` pages contain extensive hardcoded default profile data (e.g., `defaultProfiles`, `defaultLeaderboard`) that are used when the backend API returns empty or fails. This means the UI always shows fake data unless a real analysis has been run. The backend `model_profiles.py` route also has a fallback mechanism (`_get_default_profile_list`) that returns hardcoded data. This creates a misleading user experience where the system appears to have model rankings and scores that are not based on actual performance.
